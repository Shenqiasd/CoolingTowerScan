# HVAC 调研评估方案模块接入平台设计

Date: 2026-04-14

## 1. 目标

本文定义现有“调研/评估/方案”系统接入暖通空调节能项目前合同生命周期平台的方案。

接入目标不是把现有系统直接扩展成总平台，而是将其沉淀为平台内的专业子系统，承接正式项目之后的技术作业流程：

- 数据收资
- AI 识别
- 数据审核
- 能效评估
- 方案报告生成

平台本体继续负责：

- 技术初筛池
- 正式线索
- 正式项目
- gate/审批/移交
- EPC/EMC 商务分叉
- 投标
- 合同
- 文档归档与版本冻结

## 2. 当前系统定位

### 2.1 后端定位

现有后端 `hvac-energy-saving-cal-dev` 是 Spring Boot 域服务，具备以下能力：

- 项目基础资料管理
- 冷站管理
- 文件上传与存储
- AI 图片/文件识别
- 识别结果审核
- 能效评估计算
- 方案报告生成与导出

其当前数据主轴是：

- `ProjectBasicInfo`
- `ColdStation`
- `StationFile`
- `StationDevice`
- `StationOperation`
- `Evaluation`
- `SchemeReport`

### 2.2 前端定位

现有前端 `ui-sichuan-agent` 是独立 React 应用，已经形成完整业务闭环：

- 项目总览
- 数据收资
- 数据审核
- 能效评估
- 方案生成
- 报告预览/导出

它当前自带：

- 登录页
- 左侧菜单
- 独立路由壳
- 独立 token 会话

因此它现在是“独立业务子系统前端”，而不是“平台内嵌业务模块前端”。

## 3. 接入定位

推荐定位：

- **主平台 = 系统主干**
- **现有调研系统 = 专业作业子系统**

推荐接入方式：

- **前端并入平台壳**
- **后端保留为独立域服务**

这是一个“平台统一入口 + 子域独立服务 + 状态双向同步”的方案。

## 4. 业务边界

### 4.1 平台负责

- 企业主数据
- 站点主数据
- 扫描候选承接
- 正式线索管理
- 正式项目骨架
- 项目阶段 gate
- 审批与退回
- 移交包
- EPC/EMC 商务分叉
- 投标与合同
- 正式版本与归档

### 4.2 子系统负责

- 项目技术画像
- 冷站及设备作业对象
- 文件采集与识别
- 数据审核
- 能效评估
- 方案报告内容生成

### 4.3 必须剥离出子系统的内容

现有 `ProjectBasicInfo` 同时承担：

- 项目主对象
- 联系信息
- 地点信息
- 技术参数
- 阶段状态
- 收资进度

接入后必须拆分为两层：

1. 平台主项目骨架
2. 子系统技术画像对象

子系统不再拥有“正式项目”的唯一主权。

## 5. 数据架构

### 5.1 平台主键

平台侧新增并主控：

- `enterprise_id`
- `site_id`
- `formal_project_id`

### 5.2 子系统映射键

子系统新增并维护：

- `platform_project_id`
- `enterprise_id`
- `site_id`
- `research_project_status`
- `sync_version`
- `source_stage`

### 5.3 数据所有权

平台主数据拥有权：

- 企业名称
- 企业分组
- 站点名称
- 站点地址
- 正式项目编号
- 项目阶段
- 审批状态

子系统拥有权：

- 冷站
- 上传文件
- AI 识别结果
- 审核结果
- 能效评估过程数据
- 评估结果
- 报告结构数据

### 5.4 快照策略

为避免平台主数据变更影响历史报告，子系统保留快照字段：

- 项目名称快照
- 地址快照
- 联系人快照
- 电价快照
- 技术参数快照

报告生成时使用快照，而不是实时主数据。

## 6. 流程接入

### 6.1 主流程

平台流程：

1. 技术初筛池
2. 正式线索
3. 正式项目
4. 调研评估方案
5. 商务方案冻结
6. 投标
7. 合同

### 6.2 子系统子流程

调研评估方案阶段内部拆为：

1. 数据收资
2. 数据审核
3. 能效评估
4. 方案生成

### 6.3 状态同步原则

平台不直接使用子系统的 `projectStage=1/2/3` 作为平台阶段。

平台只消费子系统的专业状态摘要：

- `research_not_started`
- `research_in_progress`
- `research_submitted`
- `research_rejected`
- `research_approved`

子系统内部可以继续保留自身作业阶段，但它只能表示子域作业进度。

## 7. 接口设计

### 7.1 平台到子系统

平台向子系统提供以下操作：

- 创建调研项目
- 更新调研项目上下文
- 锁定或解锁调研项目
- 查询调研进度
- 查询评估摘要
- 查询报告清单

建议接口：

- `POST /integration/platform/projects`
- `PUT /integration/platform/projects/{platformProjectId}`
- `POST /integration/platform/projects/{platformProjectId}/lock`
- `GET /integration/platform/projects/{platformProjectId}/status`
- `GET /integration/platform/projects/{platformProjectId}/summary`
- `GET /integration/platform/projects/{platformProjectId}/reports`

### 7.2 子系统到平台

子系统向平台回传以下事件：

- 收资完成
- 审核完成
- 评估提交
- 方案生成
- 报告导出
- 方案退回

建议事件：

- `research.collection.completed`
- `research.review.completed`
- `research.evaluation.submitted`
- `research.scheme.generated`
- `research.scheme.exported`
- `research.flow.returned`

### 7.3 第一阶段同步策略

V1 推荐先做 API 拉模式：

- 平台主动拉取状态
- 平台操作主动推送创建/更新

V2 再增加事件驱动：

- 子系统异步推送关键节点事件

## 8. 前端接入方案

### 8.1 推荐模式

推荐模式：**平台壳 + 子模块页面接入**

具体做法：

- 平台统一登录
- 平台统一菜单
- 平台统一项目上下文
- 子系统页面以业务模块方式挂载

### 8.2 不推荐模式

不推荐长期使用 iframe 或完全双系统跳转，因为会导致：

- 体验割裂
- 权限重复维护
- 项目上下文断裂
- 待办与审批无法统一

### 8.3 前端改造目标

现有前端要完成三类改造：

1. 去掉独立登录依赖
2. 去掉独立左侧菜单壳
3. 改为接收平台传入的项目上下文

## 9. 权限与认证

### 9.1 认证原则

认证统一到平台，不保留子系统独立登录入口作为正式入口。

子系统接收：

- 平台 JWT
- 平台用户 ID
- 平台角色信息
- 平台项目上下文

### 9.2 授权原则

平台负责粗粒度授权：

- 谁可以进入调研模块
- 谁可以提交评估
- 谁可以发起方案冻结

子系统负责细粒度作业授权：

- 谁可以上传
- 谁可以审核
- 谁可以编辑评估设备
- 谁可以生成报告

## 10. 文档与版本

### 10.1 子系统输出物

子系统输出：

- 识别文件
- 审核结果
- 评估结果
- 报告 JSON
- 导出的 PDF/PPT/Word

### 10.2 平台归档

平台应归档：

- 调研提交版
- 评估提交版
- 方案冻结版

正式冻结动作必须由平台触发，不能只依赖子系统“已生成报告”状态。

## 11. 分期实施建议

### 11.1 一期：快速接入

- 保留现有后端域服务
- 保留现有主要业务页面
- 平台统一登录
- 平台创建项目并映射到子系统
- 平台读取子系统状态

### 11.2 二期：流程纳管

- 平台接管 gate
- 平台接管审批
- 子系统增加退回与锁定能力
- 文档归档进入平台文档中心

### 11.3 三期：深度融合

- 拆分 `ProjectBasicInfo`
- 子系统沉淀为标准技术作业服务
- 引入事件同步
- 支持方案冻结版回挂投标与合同阶段

## 12. 决策结论

结论如下：

- 不把现有系统扩成总平台
- 把现有系统接入为“调研评估方案子系统”
- 前端逐步并入平台壳
- 后端保留独立域服务
- 平台主控项目骨架、流程和正式版本
- 子系统主控专业作业过程与专业数据

该方案兼顾：

- 现有成果复用
- 上线速度
- 后续平台治理能力
- 长期可演进性
